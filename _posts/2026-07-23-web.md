Spring Boot(MSA) + React로 개발 중인 프로젝트를 로컬 개발 환경에서 실제 운영 환경으로 배포하기까지 헷갈렸던 개념들을 정리해봤다.   
MSA 프로젝트는 실행부터 배포까지 모놀리식과 다른 점이 많았다. 이번 글에서는 그 내용을 중심으로 정리해보려고 한다.


# MSA 구조 파악하기
운영 배포를 준비하기 전에, 먼저 지금 프로젝트가 어떤 구조로 나뉘어 있는지부터 명확히 해야 한다.

- 서비스가 몇 개로 나뉘어 있는지
- 서비스 간 통신 방식은 무엇인지 (동기 REST? 비동기 메시지 큐?)
- 프론트가 백엔드를 어떻게 호출하는지 (직접? Gateway를 통해서?)

모놀리식은 하나의 애플리케이션만 실행하면 되지만, MSA는 여러 서비스를 각각 실행하고 배포해야 한다. 따라서 먼저 시스템이 어떤 서비스들로 구성되어 있는지 파악하는 것이 가장 중요하다.
<br>
<br>

## 서비스 간 통신 방식
MSA에서 서비스끼리 통신하는 방식은 크게 두 가지다.
### REST API 호출 (동기)
서비스 A가 HTTP 요청을 통해 서비스 B를 직접 호출하고 응답을 기다리는 방식으로   
"재고 있어?" 처럼 즉시 답이 필요한 조회성 작업에 적합하다.
```
[주문 서비스] ---  HTTP 요청 ---> [재고 서비스]
[주문 서비스] <--- 응답 대기  --- [재고 서비스]
```

### 메시지 큐 (Kafka/RabbitMQ, 비동기)
서비스 A가 서비스 B를 직접 호출하지 않고 중간에 있는 "메시지 큐"에 메시지만 던져놓는 방식으로 수신 측 서비스가 지금 당장 살아있지 않아도 나중에 큐에 쌓인 메시지를 꺼내 처리하면 된다.
```
[주문 서비스] ---> [메시지 큐(RabbitMQ)] ---> [재고 서비스]
(메시지 던지고 바로 다음 작업 진행)            (나중에 메시지 꺼내서 처리)
```


현재 프로젝트에서는 메일 발송을 비동기로 처리하고 있다.

1. user-service/portal-service에서 메일 발송 이벤트를 RabbitMQ에 발행
2. batch-service가 해당 메시지를 소비(Consume)
3. 실제 메일 전송 수행

즉, 사용자 요청은 메일 전송이 끝날 때까지 기다리지 않고 바로 응답할 수 있다.

<br>

> 💡 **Kafka vs RabbitMQ** <br>
> **RabbitMQ**: 메시지를 큐에 넣고 소비자가 꺼내가면 사라짐 <br>
> **Kafka**: 메시지를 로그처럼 계속 쌓아두고, 여러 소비자가 각자 원하는 시점에 읽어갈 수 있음 (대용량 이벤트 스트리밍에 강함)

## API Gateway
MSA에서는 서비스마다 주소와 포트가 서로 다르다. 만약 프론트엔드가 각 서비스를 직접 호출한다면 다음과 같은 문제가 발생한다.

- 서비스 주소가 바뀔 때마다 프론트 코드도 수정해야 함
- 서비스마다 도메인이 다르면 브라우저 CORS 문제 발생

이를 해결하는 게 API Gateway다. 프론트는 하나의 진입점으로만 요청을 보내고, Gateway가 경로(path) 기준으로 실제 백엔드 서비스에 요청을 전달한다.

```
             Browser
                │
                ▼
         https://example.com
                │
          +-------------+
          |    Nginx    |
          +-------------+
        /        |        \
       ▼         ▼         ▼
 user-service portal-service board-service
 ```

```
[React 프론트] 
      |
      v
[nginx] (하나의 진입점, 예: 80/443 포트)
      |
      +--- location /user-service/   ---> proxy_pass http://user-service:8081
      +--- location /portal-service/ ---> proxy_pass http://portal-service:8082
      +--- location /board-service/  ---> proxy_pass http://board-service:8083
      +--- location /               ---> React 정적 파일 서빙 (또는 별도 서버로 proxy)
```

### API GateWay가 필요한 이유
- 서비스마다 도메인이 다르면 브라우저가 CORS를 막음
- 인증(JWT), 로깅 등을 한 곳에서 처리 가능
- 로드 밸런싱이나 SSL 종료도 Gateway에서 처리 가능
