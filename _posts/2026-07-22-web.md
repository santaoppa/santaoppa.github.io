개발을 하다 보면 **Web Server**, **WAS**, **Nginx**, **Tomcat** 같은 용어를 자연스럽게 사용한다.

하지만 막상

> "Web Server랑 WAS가 정확히 뭐가 다른데?"

라는 질문을 받으면 설명하기가 쉽지 않았다. 이 글은 인프라를 거의 모르던 내가 하나씩 개념을 정리하면서 이해한 내용을 기록한 글이다.
<br>
<br>

## 서버라는 말이 헷갈리는 이유
개발하다 보면 흔히 **"서버에 배포한다."**, 
**"서버를 하나 더 늘리자."** 라는 말을 자주 한다. 그런데 "서버"라는 말은 사실 두 가지 다른 의미로 쓰인다.

| 구분 | 의미 |
|------|------|
| 하드웨어(Server Machine) | 실제 컴퓨터(온프레미스 서버, AWS EC2 등) |
| 서버 프로그램(Server Software) | Nginx, Apache, Tomcat처럼 요청을 받아 처리하는 프로그램 |

<br>

> 서버 = 컴퓨터일 수도 있고, 프로그램일 수도 있다.
<br>

### 하나의 컴퓨터에서 여러 서버가 동시에 실행될 수 있다

예를 들어 로컬에서 개발할 때

```
localhost:3000   → React / Next.js

localhost:8080   → Spring Boot

localhost:3306   → MySQL
```

모두 하나의 PC에서 실행된다. 컴퓨터는 하나지만, 각 프로그램이 서로 다른 포트에서 요청을 기다리고 있는 것이다.
<br>
<br>

## Web Server와 WAS는 무엇이 다를까?
- Web 서버 (Nginx, Apache): 정적 파일(HTML, CSS, JS)을 서빙하고, 요청을 뒤로 전달하는 관문 역할
- WAS (Tomcat, Spring Boot 등): 실제 비즈니스 로직을 처리하고 DB와 통신하는 곳
<br>

### Web Server의 역할

- 대표적으로 Nginx, Apache가 있다.
- Web Server는 HTML, CSS, JavaScript, 이미지같은 정적 파일을 매우 빠르게 제공한다.
- 또한 API 요청은 직접 처리하지 않고 WAS에게 전달한다.

예를 들어

```nginx
location /api/ {
    proxy_pass http://was_server:8080;
}
```

사용자가

```
GET /api/users
```

를 요청하면 Nginx는 그대로 WAS에게 전달한다.
이 기능을 Reverse Proxy라고 한다.
<br>

### WAS의 역할
- 대표적으로 Tomcat, Spring Boot 등이 있다.
- WAS는 로그인/회원가입/주문/DB 조회처럼 비즈니스 로직을 수행한다.

예를 들어

```
GET /api/products
```

```
Spring Controller 실행

↓

Service 실행

↓

DB 조회

↓

JSON 응답
```
모든 과정은 WAS에서 수행된다.

---

쉽게 비유하면

```
손님
   │
   ▼
카운터 직원(Web Server)
   │
   ▼
주방(WAS)
```

카운터 직원은

- 주문을 받고
- 전달하고
- 음식이 나오면 전달한다.

실제로 요리하는 곳은 주방이다.
<br>

### 왜 굳이 Web Server와 WAS를 나눌까?
- 정적 파일 처리가 훨씬 빠름: Nginx는 정적 파일을 매우 빠르게 제공하도록 만들어졌다. 굳이 Tomcat이 처리할 필요가 없다.
- 역할 분리: Web Server는 요청만 관리하고, WAS는 비즈니스 로직에만 집중한다.
- 확장성: 트래픽이 늘어나면 WAS만 여러 대 늘리면 된다.
<br>
<br>

## 로컬 개발 환경과 운영 환경의 차이
로컬에서는 보통 PC 한 대 안에 Web 서버, WAS, DB를 (포트만 다르게) 전부 띄워서 쓴다. 반면 운영 환경에서는 이 세가지를 물리적으로 (또는 클라우드에서 논리적으로) 분리한 컴퓨터에 각각 올린다.
<br>

### 왜 분리할까?
- 자원 경쟁 방지: DB가 무거운 쿼리로 CPU를 많이 쓰면, 같은 서버에 있던 WAS까지 느려진다. 분리하면 서로 영향을 주지 않는다.
- 독립적 확장: 트래픽이 늘면 WAS만 여러 대로 늘릴 수 있다.
- 보안: DB는 외부에서 직접 접근하지 못하게 막고, Web 서버만 외부에 노출시킬 수 있다.
- 장애 격리: WAS가 죽어도 DB는 살아있어서 복구가 빠르다.
<br>
<br>

## 도메인은 어떻게 서버를 찾아갈까?
로컬에서는 hosts 파일에 가짜 도메인을 매핑해서 썼지만, 이건 내 PC에서만 통하는 임시방편이다. 운영에서는 진짜 도메인을 구입해서 DNS에 등록한다.
```
shop.mycompany.com → 52.78.123.45 (공인 IP)
```
<br>
여기서 DNS도 두 종류로 나뉜다.
- 공개 DNS: 전 세계가 공유하는 시스템. 도메인을 사면 이 시스템에 등록되어 전 세계 어디서든 같은 IP로 연결된다.
- 내부(사설) DNS: 회사 내부망(VPC)에서만 통하는 DNS. 예를 들어 was.internal 같은 이름을 WAS의 사설 IP로 매핑해두고, IP가 바뀌어도 이 이름은 그대로 쓸 수 있게 한다.

<br>

### 공인 IP와 사설 IP, 그리고 VPC
실제 클라우드 구성에서는 공인 IP와 사설 IP를 명확히 구분해서 쓴다.

- 공인 IP: 전 세계 인터넷 어디서든 접근 가능한 주소. Web 서버만 이 IP를 가진다.
- 사설 IP: 회사가 만든 내부망(VPC, Virtual Private Cloud) 안에서만 통하는 주소. WAS와 DB, Redis 등은 이 사설 IP만 가진다.
  
```
Web server:
  공인 IP  52.78.123.45   ← 외부(인터넷)에서 접근
  사설 IP  10.0.1.10      ← 내부망에서 통신

WAS #1~3, Redis:
  사설 IP만 존재            ← 외부에서 직접 접근 불가능
```

<br>
WAS는 아예 공인 IP 자체가 없다. 사용자가 WAS에 직접 접근할 이유가 없기 때문이다. 따라서 Web 서버를 거쳐야만 서비스에 접근할 수 있고 덕분에 외부에서 WAS로 직접 침입을 시도할 방법 자체가 없어진다.
<br>
<br>

## 여러 대의 WAS로 트래픽을 나누는 로드밸런서

```
         Nginx
           │
    ┌──────┼──────┐
    ▼      ▼      ▼
 WAS1   WAS2   WAS3
```

트래픽이 늘어나면 WAS를 한 대에서 여러 대로 늘린다. 그럼 Web 서버는 요청이 올 때마다 어느 WAS로 보낼지 결정을 해야 하는데, 이 역할을 로드밸런서가 한다.
<br>
Nginx 자체가 리버스 프록시 기능 안에 로드밸런싱 기능까지 포함하고 있기 때문에 upstream 블록에 WAS 서버들을 나열해두면 Nginx가 요청을 순서대로 분산스켜준다. (대용량 트래픽 환경에서는 AWS ELB같은 전용 로드밸런서 서비스를 별도로 두기도 한다.)

```nginx
upstream was_group {
    server 10.0.2.11:8080;
    server 10.0.2.12:8080;
    server 10.0.2.13:8080;
}

location /api/ {
    proxy_pass http://was_group;
}
```

<br>

### 여러 대의 WAS에서 로그인은 어떻게 유지될까?
만약 WAS 메모리에만 세션을 저장한다면

```
로그인

↓

WAS2 저장

↓

다음 요청

↓

WAS3

↓

세션 없음
```
<br>

각 WAS가 세션 정보를 자기 서버 메모리에만 저장하기 때문에 생기는 문제다. 해결 방법은 크게 세 가지다.

- Sticky Session: 특정 사용자를 항상 같은 WAS로만 보내도록 고정. 구현은 간단하지만 그 WAS에 트래픽이 몰리거나, 죽으면 그 사용자들 전부 로그아웃되는 단점이 있다.   
- 세션 클러스터링: WAS끼리 세션 정보를 실시간으로 복제. 서버가 많아질수록 복제 트래픽 부담이 커진다.   
- 세션을 외부 저장소(Redis)로 분리: 세션을 WAS 메모리가 아니라 별도의 공용 저장소에 저장. 어느 WAS로 요청이 가든 Redis에 물어보면 되기 때문에 가장 널리 쓰이는 방식이다.

<br>

```
      WAS1

        │

Redis

        │

      WAS2
```

> 어느 WAS로 가든 → Redis에서 같은 세션 정보를 조회

<br>

## HTTPS와 SSL인증서
사용자와 서버 사이의 통신은 기본적으로 평문(HTTP)이라 중간에 가로채이면 내용이 그대로 노출된다. 이를 암호화한 것이 HTTPS다.

암호화만으로는 부족한 부분도 있다. "지금 접속한 이 서버가 진짜 그 회사가 맞는지"를 증명해야 하는데, 이걸 보증해주는 것이 인증서다. 인증기관(CA)이 "이 도메인은 이 회사 것이 맞다"고 보증한 디지털 증명서를 Web 서버에 설치해두면, 브라우저가 이를 검증하고 암호화 통신을 시작한다.

일반적으로

- HTTP → 80
- HTTPS → 443

포트를 사용한다.

도메인이 하나면 인증서도 하나면 충분하지만, 프론트엔드/백엔드 도메인을 분리하면 인증서도 각각 필요해진다.
외부와 통신하는 구간(사용자-Web 서버)만 HTTPS로 암호화하고, 내부망(Web-WAS) 구간은 어차피 사설망이라 HTTP로 두는 선택지도 있다.
<br>
<br>

## 전체 그림으로 정리
지금까지 나온 개념을 하나의 그림으로 정리하면 다음과 같다.

![alt text](/assets/images/posts_img/image.png)

1. 사용자가 도메인(shop.mycompany.com) 입력
2. DNS가 이 도메인을 Web 서버의 공인 IP로 변환
3. 사용자가 HTTPS(443)로 Web 서버에 접속, 이 과정에서 인증서로 신원 검증
4. Web 서버(Nginx)가 요청을 받아 정적 파일은 직접 응답하고, API 요청은 로드밸런싱을 거쳐 여러 WAS 중 하나(사설 IP)로 전달
5. WAS가 세션 확인이 필요하면 Redis(사설 IP)에서 세션 정보를 조회 — 어느 WAS로 가든 동일한 결과
6. WAS가 로직을 처리하고 DB(역시 사설망 안)와 통신해 응답을 돌려줌
